# B07 · Build Your Own 搜索、约束与规划引擎

- **对应课程：**[AI02](../curriculum/modules/07-classical-ai-search-planning.md)
- **建议投入：**40-60 小时
- **难度：**基础
- **学习方法：**每一关都完成“构建 → 验证 → 解释”，再进入下一关

## 最终要造什么

一个可以表达状态、求解路径、约束和博弈决策的经典 AI 工具箱。

## 第一遍约束

不使用现成搜索/约束库完成核心算法。

这些约束只用于暴露原理。通过核心验收后，应与成熟库或框架对照，不建议把教学实现直接用于高风险生产。

## 逐步关卡

| 关卡 | 亲手构建 | 通过检查 |
| --- | --- | --- |
| 1. 建问题接口 | 定义 state、actions、transition、goal 和 cost。 | 同一接口可承载至少两个问题。 |
| 2. 无信息搜索 | 实现 BFS、DFS、uniform-cost 和访问集。 | 比较完备性、最优性和内存。 |
| 3. 启发式搜索 | 实现 greedy 与 A*，设计可采纳启发式。 | 对比扩展节点和路径代价。 |
| 4. 约束求解 | 实现变量、域、约束、回溯和前向检查。 | 可解释无解原因或冲突。 |
| 5. 博弈决策 | 实现 minimax、评估函数和 alpha-beta。 | 不同深度下行为与耗时可测。 |
| 6. 混合方案 | 对同一业务任务比较规则、搜索和 LLM。 | 用正确性、成本、可解释性做选型。 |

## 通用验收

- 从空目录或最小脚手架开始，提交每一关的独立 commit。
- 正常、边界、错误和故障案例都有自动测试或可重复实验。
- 能画出输入、状态、处理、输出和失败路径。
- 能闭卷解释一个关键算法或协议，并现场修改一个小行为。
- 记录 AI 提出的关键建议、人工采纳/拒绝理由和最终验证。
- 与一个成熟实现比较正确性、性能、功能边界和维护成本。

## AI 使用规则

可以让 AI 解释概念、生成测试想法、审查代码和定位错误；不得让 AI 一次性生成最终仓库后直接提交。每关先写自己的接口、假设和验收，再让 AI 参与实现。

## 完成证据

1. 代码仓库与阶段提交。
2. 架构/算法说明。
3. 测试或实验报告。
4. 至少三个失败样本。
5. 成熟实现对照。
6. 10-20 分钟口试记录。
