# B02 · Build Your Own HTTP 与完整 Web 服务

- **对应课程：**[SE02](../curriculum/modules/02-web-api-databases.md)
- **建议投入：**45-65 小时
- **难度：**基础
- **学习方法：**每一关都完成“构建 → 验证 → 解释”，再进入下一关

## 最终要造什么

从原始 HTTP 请求处理器成长为带前端、API、数据库和文档的 Web 应用。

## 第一遍约束

前三关不使用 Web 框架；先亲手观察请求行、header、body 和响应。

这些约束只用于暴露原理。通过核心验收后，应与成熟库或框架对照，不建议把教学实现直接用于高风险生产。

## 逐步关卡

| 关卡 | 亲手构建 | 通过检查 |
| --- | --- | --- |
| 1. 捕获 HTTP | 用 socket 接收请求并保存原始字节。 | 能逐段解释浏览器发来的内容。 |
| 2. 写解析与路由 | 解析方法、路径、查询和 JSON，映射到处理函数。 | 错误请求返回明确 4xx，而非崩溃。 |
| 3. 定义资源契约 | 设计 CRUD、状态码、验证和错误结构。 | 接口示例先于数据库实现。 |
| 4. 加持久化 | 用 SQL 建表、约束、事务、索引和迁移。 | 并发或中途失败不会留下半条业务记录。 |
| 5. 接前端流程 | 完成表单、列表、编辑、加载、空态和错误态。 | 可以从一次点击追踪到数据库。 |
| 6. 框架化对照 | 迁移到 FastAPI，比较自实现与框架负责的边界。 | 自动接口文档、集成测试和端到端流程通过。 |

## 通用验收

- 从空目录或最小脚手架开始，提交每一关的独立 commit。
- 正常、边界、错误和故障案例都有自动测试或可重复实验。
- 能画出输入、状态、处理、输出和失败路径。
- 能闭卷解释一个关键算法或协议，并现场修改一个小行为。
- 记录 AI 提出的关键建议、人工采纳/拒绝理由和最终验证。
- 与一个成熟实现比较正确性、性能、功能边界和维护成本。

## AI 使用规则

可以让 AI 解释概念、生成测试想法、审查代码和定位错误；不得让 AI 一次性生成最终仓库后直接提交。每关先写自己的接口、假设和验收，再让 AI 参与实现。

## 完成证据

1. 代码仓库与阶段提交。
2. 架构/算法说明。
3. 测试或实验报告。
4. 至少三个失败样本。
5. 成熟实现对照。
6. 10-20 分钟口试记录。
