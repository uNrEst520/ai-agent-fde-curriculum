# 面试故事与表达

## 1. 从建筑设计师到 AI Builder

我不是传统 CS 背景，但建筑设计训练让我擅长处理复杂约束、流程拆解和用户体验。AI 工具成熟后，我把这种能力迁移到软件产品：用 Codex、MCP、N8N、Opal 和本地 Agent 工作流，把想法快速变成可试用工具。

## 2. 为什么选择网球场景

网球训练有明确闭环：比赛暴露问题，训练修正问题，视频验证动作，战术解释选择。这比做一个通用聊天机器人更能展示垂直场景理解和产品判断。

## 3. 主项目怎么体现工程能力

“随身网球教练 MVP”不是静态原型。它有前端、后端、视频上传、转码、姿态分析、报告生成、导出包、测试脚本和参考素材库。新增的训练处方把视觉证据转成下一步训练，展示了从数据到行动的工程闭环。

## 4. Codex / AI 辅助开发怎么用

我的工作方式是先定义产品闭环和验收标准，再让 Codex 辅助读代码、写模块、补测试和做 smoke 验证。我不把 AI 当“自动写代码机器”，而是当一个可调用文件、工具和测试环境的工程搭档。

## 5. 创业公司为什么需要这种能力

早期团队不一定需要一个只会写某个模块的工程师，更需要能把业务、AI 能力、原型、数据和交付串起来的人。我能在较短时间内验证一个垂直 AI 产品是不是成立，并把它做成可演示、可迭代的版本。

## 6. 下一步怎么扩展

下一步不是盲目训练大模型，而是先把四个项目连成数据闭环：记分复盘产生问题，训练计划安排任务，视频识别验证动作，战术板解释打法。等闭环稳定后，再接入 LLM 做追问、总结和个性化教练话术。

## 高频问答

**你不是传统程序员，怎么胜任 Agent 工程岗位？**  
我不把自己包装成底层算法或传统全栈工程师，而是 AI Builder。我的优势是能用 AI 工具链和工程化方式把业务流程做成可用产品，并且知道如何测试、迭代和展示价值。

**这个项目是不是只靠 AI 生成代码？**  
不是。AI 参与了实现效率，但项目本身有清晰架构、测试、报告导出、数据结构和产品闭环。我能解释每个模块的作用、取舍和下一步。

**为什么不用大模型直接生成所有建议？**  
第一阶段先用规则 Agent，因为它便宜、稳定、可解释。等视觉证据和训练库稳定后，再让 LLM 做语言表达、多轮追问和个性化计划。

**如果去创业公司，你能带来什么？**  
我能把创始人的模糊想法快速拆成 AI 产品原型，接入现有工具和 API，做出可试用 Demo，并用真实用户反馈决定下一步，而不是停留在概念 PPT。
