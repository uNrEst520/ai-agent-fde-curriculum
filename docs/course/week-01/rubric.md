# Week 01 批改标准

满分 100 分。本周不按“代码写得多不多”评分，而按“是否摆脱 bypass、是否能解释和验收”评分。

## 1. 业务理解，20 分

- 0-8 分：只描述功能，没有说清用户和问题。
- 9-15 分：能说明用户、场景和基本价值。
- 16-20 分：能把项目放进真实网球训练/教练/机构工作流，并说明为什么值得做。

## 2. 工程地图，25 分

- 0-10 分：只列文件名，不知道各自负责什么。
- 11-18 分：能说明前端、后端、报告、测试、脚本的基本分工。
- 19-25 分：能把文件分工和“用户到报告”的链路连起来讲清楚。

## 3. 命令理解，15 分

- 0-5 分：只复制命令，不知道命令在做什么。
- 6-11 分：能解释 `cd`、`npm run build`、`pytest` 的大致作用。
- 12-15 分：能说明每条命令对应的验收对象和失败时应先看哪里。

## 4. 测试与验收，15 分

- 0-5 分：没有运行测试或没有记录结果。
- 6-11 分：运行了测试，知道通过/失败。
- 12-15 分：能说明测试保护了哪个功能，以及为什么这能减少 AI bypass 风险。

## 5. 小改动复盘，15 分

- 0-5 分：只让 Codex 改了，没有说明原因。
- 6-11 分：能说明改动内容、涉及文件和验证结果。
- 12-15 分：能说明业务理由、影响范围、副作用和回滚/修正思路。

## 6. FDE 式上岗思考，10 分

- 0-4 分：只说“以后可以商业化”。
- 5-8 分：能提出真实部署需要补的流程、数据、培训或验收。
- 9-10 分：能从企业采用角度说明 AI 如何从 Demo 变成日常工作岗位的一部分。

## 批改时我会重点看什么

我会优先批改这些点：

1. 你有没有把不懂的问题说具体。
2. 你有没有把业务链路和工程链路分开讲清楚。
3. 你有没有拿命令输出、测试结果、页面观察作为证据。
4. 你有没有意识到 AI 写代码后，人仍然要负责验收。
5. 你有没有开始形成自己的表达，而不是复制我的话。

## 本周优秀作业标准

优秀作业不一定长，但一定清楚。理想状态是：我读完后相信你能在面试中用 5 分钟讲明白这个项目，也相信你下次让 Codex 改功能时，知道该看哪里、跑什么、怎么判断对错。
