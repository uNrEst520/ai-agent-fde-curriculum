# SE03 · AI 辅助软件工程

- **建议时长：**6 周 · 每周 12-15 小时
- **先修课程：**SE01
- **课程性质：**主线课
- **验收方式：**构建项目 + 交付任务 + 口试

## 为什么学

AI 可以快速生成代码，但需求边界、仓库理解、变更控制、测试策略和发布责任仍属于工程师。本课训练如何指挥和审查 Coding Agent。

## 学习目标

- 把需求转成可观察验收标准和最小变更
- 在陌生代码库中定位所有权、依赖与测试
- 审查 AI 生成代码并用自动化证据控制回归

## 周计划

| 周 | 主题与实践 |
| --- | --- |
| 1 | Git、提交、分支、差异与可逆变更 |
| 2 | 代码库地图、搜索、调用链和依赖阅读 |
| 3 | 需求拆解、接口设计与变更计划 |
| 4 | 单元、集成、端到端测试与测试替身 |
| 5 | 代码审查、静态检查、CI 和供应链风险 |
| 6 | 用 Coding Agent 完成一次受控功能交付 |

## 从零构建

构建一个小型代码任务执行器，记录需求、计划、文件改动、验证命令和审查结论。

## 真实交付

在陌生仓库完成一个有明确验收标准的功能，提交设计说明、测试、差异审查和回滚方案。

## 主要资源

- [The Missing Semester](https://missing.csail.mit.edu/)
- [Pro Git](https://git-scm.com/book/en/v2)
- [GitHub Actions 文档](https://docs.github.com/en/actions)

资源用于建立主线，最新论文与工具变化从 [研究雷达](../../../research/README.md) 进入候选，不会未经验证直接替换稳定原理。

## 必交证据

- 一份代码库工程地图
- AI 建议被拒绝或修正的证据
- 一次 CI 失败到修复的完整记录

## 反 bypass 口试

1. 为什么增加测试数量不等于降低风险？
2. 如何判断一个 AI 建议超出了需求范围？
3. 单元测试和端到端测试分别保护什么？
