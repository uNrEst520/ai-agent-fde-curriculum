# B01 · Build Your Own 命令行数据应用

- **对应课程：**[SE01](../curriculum/modules/01-python-computational-thinking.md)
- **建议投入：**35-50 小时
- **难度：**入门
- **学习方法：**每一关都完成“构建 → 验证 → 解释”，再进入下一关

## 最终要造什么

一个可读取、校验、转换、查询并导出数据的命令行工具。

## 第一遍约束

核心逻辑只用 Python 标准库；前四关不使用 pandas、click 或数据库 ORM。

这些约束只用于暴露原理。通过核心验收后，应与成熟库或框架对照，不建议把教学实现直接用于高风险生产。

## 逐步关卡

| 关卡 | 亲手构建 | 通过检查 |
| --- | --- | --- |
| 1. 写行为契约 | 定义三种输入、五个命令、输出格式和错误码。 | 先写 10 个示例，不写实现。 |
| 2. 解析输入 | 读取 CSV/JSON，处理编码、空值、重复和错误行。 | 坏数据不会让程序静默产生错误结果。 |
| 3. 建数据模型 | 使用函数、dataclass 和类型表达记录与规则。 | 规则测试不依赖文件系统。 |
| 4. 实现查询转换 | 完成过滤、分组、排序、统计和导出。 | 同一输入重复运行结果一致。 |
| 5. 写最小测试器 | 先用 assert 和临时目录实现测试运行，再接 pytest。 | 正常、边界、异常至少 30 个案例。 |
| 6. 打包交付 | 加入配置、日志、帮助、版本和可安装入口。 | 新环境能按 README 在 10 分钟内运行。 |

## 通用验收

- 从空目录或最小脚手架开始，提交每一关的独立 commit。
- 正常、边界、错误和故障案例都有自动测试或可重复实验。
- 能画出输入、状态、处理、输出和失败路径。
- 能闭卷解释一个关键算法或协议，并现场修改一个小行为。
- 记录 AI 提出的关键建议、人工采纳/拒绝理由和最终验证。
- 与一个成熟实现比较正确性、性能、功能边界和维护成本。

## AI 使用规则

可以让 AI 解释概念、生成测试想法、审查代码和定位错误；不得让 AI 一次性生成最终仓库后直接提交。每关先写自己的接口、假设和验收，再让 AI 参与实现。

## 完成证据

1. 代码仓库与阶段提交。
2. 架构/算法说明。
3. 测试或实验报告。
4. 至少三个失败样本。
5. 成熟实现对照。
6. 10-20 分钟口试记录。
