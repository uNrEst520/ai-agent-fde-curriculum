# DE01 · 面向 AI 的数据工程

- **建议时长：**5 周 · 每周 12-15 小时
- **先修课程：**SE02、SE03
- **课程性质：**主线课
- **验收方式：**构建项目 + 交付任务 + 口试

## 为什么学

模型输出质量受数据来源、版本、血缘和更新方式约束。没有可追踪的数据管道，RAG、训练和评测都无法可靠复现。

## 学习目标

- 设计采集、清洗、验证、转换、存储和版本流程
- 定义数据契约、质量检查与增量更新策略
- 追踪一条输出回到原始数据和处理版本

## 周计划

| 周 | 主题与实践 |
| --- | --- |
| 1 | 数据格式、模式、质量维度与数据契约 |
| 2 | 批处理、流处理、幂等和增量计算 |
| 3 | SQL 分析、列式存储与分区 |
| 4 | 编排、血缘、版本、隐私和权限 |
| 5 | 构建可重跑的数据产品与质量看板 |

## 从零构建

[进入逐步构建指南](../../builds/reproducible-data-pipeline.md)

用 Python、SQL 和本地文件构建一个具备校验、去重、增量更新和血缘记录的数据管道。

## 真实交付

将管道接入真实或脱敏数据源，加入调度、失败恢复、质量告警和数据说明书。

## 主要资源

- [DuckDB 文档](https://duckdb.org/docs/)
- [dbt Fundamentals](https://docs.getdbt.com/docs/introduction)
- [Apache Airflow 文档](https://airflow.apache.org/docs/)

资源用于建立主线，最新论文与工具变化从 [研究雷达](../../../research/README.md) 进入候选，不会未经验证直接替换稳定原理。

## 必交证据

- 数据契约和血缘图
- 重复运行结果一致的证明
- 脏数据与模式变化故障记录

## 反 bypass 口试

1. 幂等为什么对数据管道重要？
2. 数据版本与代码版本如何共同决定结果？
3. 什么情况下应该拒绝让下游继续运行？
