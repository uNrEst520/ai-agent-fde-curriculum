# AP01 · 模型 API、结构化输出与 AI 体验

- **建议时长：**4 周 · 每周 12-15 小时
- **先修课程：**SE02、ML01
- **课程性质：**主线课
- **验收方式：**构建项目 + 交付任务 + 口试

## 为什么学

把模型接进产品不是“发一个 prompt”。需要处理上下文、结构化数据、流式体验、错误、成本、隐私和人工确认。

## 学习目标

- 实现供应商无关的模型调用边界
- 设计结构化输出、状态管理和人机交互
- 用基线任务和失败案例选择模型与提示策略

## 周计划

| 周 | 主题与实践 |
| --- | --- |
| 1 | 模型 API、消息、token、采样、流式与成本 |
| 2 | 提示、结构化输出、验证、重试和降级 |
| 3 | AI 体验：等待、编辑、确认、引用和反馈 |
| 4 | 多模型对照、任务评测与完整应用 |

## 从零构建

[进入逐步构建指南](../../builds/model-client-runtime.md)

不使用 Agent 框架，构建带消息状态、结构验证、重试、超时、缓存和成本记录的模型客户端。

## 真实交付

完成一个面向明确用户任务的 AI Web 功能，并用规则基线、至少两种模型和人工评审对照。

## 主要资源

- [OpenAI API 文档](https://platform.openai.com/docs/)
- [Anthropic API 文档](https://docs.anthropic.com/)
- [Google Gemini API 文档](https://ai.google.dev/gemini-api/docs)

资源用于建立主线，最新论文与工具变化从 [研究雷达](../../../research/README.md) 进入候选，不会未经验证直接替换稳定原理。

## 必交证据

- 模型与规则基线对照
- 结构化输出失败样本
- 成本、延迟和用户体验权衡记录

## 反 bypass 口试

1. 什么时候结构化输出仍然不可信？
2. 流式输出改善了什么，又增加了什么复杂度？
3. 模型切换为什么不应散落在业务代码中？
