# B05 · Build Your Own 可复现数据管道

- **对应课程：**[DE01](../curriculum/modules/05-data-engineering.md)
- **建议投入：**35-55 小时
- **难度：**基础
- **学习方法：**每一关都完成“构建 → 验证 → 解释”，再进入下一关

## 最终要造什么

一个可增量重跑、具备数据契约、质量门禁、血缘和失败恢复的管道。

## 第一遍约束

先用 Python、SQL 和本地列式文件实现，再接编排工具。

这些约束只用于暴露原理。通过核心验收后，应与成熟库或框架对照，不建议把教学实现直接用于高风险生产。

## 逐步关卡

| 关卡 | 亲手构建 | 通过检查 |
| --- | --- | --- |
| 1. 定义数据契约 | 列出字段、类型、主键、允许值、时效和隐私级别。 | 错误样本能明确归因到契约规则。 |
| 2. 建原始层 | 原样保存来源、时间、校验和与批次 id。 | 任意结果可回到原始输入。 |
| 3. 清洗标准化 | 处理缺失、重复、单位、时区和 schema 变化。 | 转换函数对同一输入幂等。 |
| 4. 加质量门禁 | 检查完整性、唯一性、分布和新鲜度。 | 故意注入脏数据会阻止发布。 |
| 5. 做增量更新 | 使用水位、分区或变更记录只处理新增数据。 | 全量与增量结果一致。 |
| 6. 加血缘编排 | 记录代码版、数据版、运行状态和下游影响。 | 失败后可从安全检查点恢复。 |

## 通用验收

- 从空目录或最小脚手架开始，提交每一关的独立 commit。
- 正常、边界、错误和故障案例都有自动测试或可重复实验。
- 能画出输入、状态、处理、输出和失败路径。
- 能闭卷解释一个关键算法或协议，并现场修改一个小行为。
- 记录 AI 提出的关键建议、人工采纳/拒绝理由和最终验证。
- 与一个成熟实现比较正确性、性能、功能边界和维护成本。

## AI 使用规则

可以让 AI 解释概念、生成测试想法、审查代码和定位错误；不得让 AI 一次性生成最终仓库后直接提交。每关先写自己的接口、假设和验收，再让 AI 参与实现。

## 完成证据

1. 代码仓库与阶段提交。
2. 架构/算法说明。
3. 测试或实验报告。
4. 至少三个失败样本。
5. 成熟实现对照。
6. 10-20 分钟口试记录。
