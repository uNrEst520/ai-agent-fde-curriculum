# B13 · Build Your Own 工具协议与 MCP 系统

- **对应课程：**[AP03](../curriculum/modules/13-tool-use-mcp.md)
- **建议投入：**45-65 小时
- **难度：**中级
- **学习方法：**每一关都完成“构建 → 验证 → 解释”，再进入下一关

## 最终要造什么

从自定义工具协议到 MCP 服务端/客户端、授权、确认和审计的完整连接系统。

## 第一遍约束

先实现最小 RPC 和工具执行边界，再使用官方 SDK。

这些约束只用于暴露原理。通过核心验收后，应与成熟库或框架对照，不建议把教学实现直接用于高风险生产。

## 逐步关卡

| 关卡 | 亲手构建 | 通过检查 |
| --- | --- | --- |
| 1. 工具 schema | 定义名称、说明、参数、结果、错误和副作用级别。 | 无效参数在执行前被拒绝。 |
| 2. 执行器 | 实现注册、调用、超时、取消、幂等和审计。 | 重复请求不会重复危险动作。 |
| 3. 最小 RPC | 用 JSON 消息实现 request id、method、result 和 error。 | 乱序、断连和未知方法可处理。 |
| 4. MCP 服务端 | 映射生命周期、tools/list 和 tools/call。 | 用 Inspector 或测试客户端验证。 |
| 5. 客户端与同意 | 实现发现、选择、权限、人工确认和结果回传。 | 写操作必须显式确认。 |
| 6. 威胁测试 | 测试恶意描述、参数注入、越权、数据外传。 | 形成权限矩阵和处置记录。 |

## 通用验收

- 从空目录或最小脚手架开始，提交每一关的独立 commit。
- 正常、边界、错误和故障案例都有自动测试或可重复实验。
- 能画出输入、状态、处理、输出和失败路径。
- 能闭卷解释一个关键算法或协议，并现场修改一个小行为。
- 记录 AI 提出的关键建议、人工采纳/拒绝理由和最终验证。
- 与一个成熟实现比较正确性、性能、功能边界和维护成本。

## AI 使用规则

可以让 AI 解释概念、生成测试想法、审查代码和定位错误；不得让 AI 一次性生成最终仓库后直接提交。每关先写自己的接口、假设和验收，再让 AI 参与实现。

## 完成证据

1. 代码仓库与阶段提交。
2. 架构/算法说明。
3. 测试或实验报告。
4. 至少三个失败样本。
5. 成熟实现对照。
6. 10-20 分钟口试记录。
