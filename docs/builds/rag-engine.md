# B12 · Build Your Own 从零构建 RAG 引擎

- **对应课程：**[AP02](../curriculum/modules/12-rag-knowledge-engineering.md)
- **建议投入：**45-70 小时
- **难度：**中级
- **学习方法：**每一关都完成“构建 → 验证 → 解释”，再进入下一关

## 最终要造什么

一个从文档解析到检索、引用、答案和分层评测的知识系统。

## 第一遍约束

前三关不使用向量数据库或 RAG 框架；先实现可检查的检索基线。

这些约束只用于暴露原理。通过核心验收后，应与成熟库或框架对照，不建议把教学实现直接用于高风险生产。

## 逐步关卡

| 关卡 | 亲手构建 | 通过检查 |
| --- | --- | --- |
| 1. 解析与切分 | 保留文档 id、层级、页码、时间、权限和来源。 | 每个 chunk 可回到原文位置。 |
| 2. 倒排检索 | 实现 token、词频、IDF、BM25 或简化等价。 | 查询命中可解释到具体词项。 |
| 3. 向量检索 | 实现 embedding 接口、余弦、top-k 和索引持久化。 | 小数据用暴力搜索验证近邻结果。 |
| 4. 混合与重排 | 融合词法/向量分数，加入过滤和重排。 | 按问题类型比较召回。 |
| 5. 生成与引用 | 组装上下文，要求逐条引用并允许无答案。 | 引用必须支持答案主张。 |
| 6. 评测与攻击 | 分开评检索和答案，加入过期、冲突、越权和注入。 | 至少 30 条回归集与失败分类。 |

## 通用验收

- 从空目录或最小脚手架开始，提交每一关的独立 commit。
- 正常、边界、错误和故障案例都有自动测试或可重复实验。
- 能画出输入、状态、处理、输出和失败路径。
- 能闭卷解释一个关键算法或协议，并现场修改一个小行为。
- 记录 AI 提出的关键建议、人工采纳/拒绝理由和最终验证。
- 与一个成熟实现比较正确性、性能、功能边界和维护成本。

## AI 使用规则

可以让 AI 解释概念、生成测试想法、审查代码和定位错误；不得让 AI 一次性生成最终仓库后直接提交。每关先写自己的接口、假设和验收，再让 AI 参与实现。

## 完成证据

1. 代码仓库与阶段提交。
2. 架构/算法说明。
3. 测试或实验报告。
4. 至少三个失败样本。
5. 成熟实现对照。
6. 10-20 分钟口试记录。
