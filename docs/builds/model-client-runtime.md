# B11 · Build Your Own 供应商无关模型客户端

- **对应课程：**[AP01](../curriculum/modules/11-model-api-ai-ux.md)
- **建议投入：**30-45 小时
- **难度：**中级
- **学习方法：**每一关都完成“构建 → 验证 → 解释”，再进入下一关

## 最终要造什么

一个统一管理消息、结构化输出、流式、超时、重试、成本和供应商切换的模型客户端。

## 第一遍约束

不使用 Agent 框架；业务代码不能直接依赖某个供应商响应结构。

这些约束只用于暴露原理。通过核心验收后，应与成熟库或框架对照，不建议把教学实现直接用于高风险生产。

## 逐步关卡

| 关卡 | 亲手构建 | 通过检查 |
| --- | --- | --- |
| 1. 定义接口 | 设计 request、response、usage、error 和 stream event。 | 用 fake provider 完成测试。 |
| 2. 消息状态 | 实现角色、上下文裁剪、会话和确定性序列化。 | 同一状态可重放。 |
| 3. 结构验证 | 定义 schema、解析、修复上限和拒绝策略。 | 无效输出不会进入业务数据。 |
| 4. 流式体验 | 处理 token、完成、取消、超时和部分结果。 | 用户取消能终止下游请求。 |
| 5. 可靠性与成本 | 加入有限重试、fallback、缓存、配额和用量。 | 故障不会造成无限费用。 |
| 6. 多模型对照 | 实现两个 provider 或一个真实一个 fake。 | 同一任务比较质量、延迟、成本。 |

## 通用验收

- 从空目录或最小脚手架开始，提交每一关的独立 commit。
- 正常、边界、错误和故障案例都有自动测试或可重复实验。
- 能画出输入、状态、处理、输出和失败路径。
- 能闭卷解释一个关键算法或协议，并现场修改一个小行为。
- 记录 AI 提出的关键建议、人工采纳/拒绝理由和最终验证。
- 与一个成熟实现比较正确性、性能、功能边界和维护成本。

## AI 使用规则

可以让 AI 解释概念、生成测试想法、审查代码和定位错误；不得让 AI 一次性生成最终仓库后直接提交。每关先写自己的接口、假设和验收，再让 AI 参与实现。

## 完成证据

1. 代码仓库与阶段提交。
2. 架构/算法说明。
3. 测试或实验报告。
4. 至少三个失败样本。
5. 成熟实现对照。
6. 10-20 分钟口试记录。
