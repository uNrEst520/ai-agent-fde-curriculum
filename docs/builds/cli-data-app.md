# B01 · Python 代码审阅与逻辑验证实验

- **对应课程：**[SE01](../curriculum/modules/01-python-computational-thinking.md)
- **建议投入：**25-35 小时
- **难度：**入门
- **学习方法：**每一关都完成“先预测 → 再观察 → 找证据 → 作判断”

## 最终要造什么

不是从空目录编写一个新应用，而是为一个已有 Python 项目制作一份**代码审阅与逻辑验证档案**。

档案必须能回答：

- 用户的一次操作经过了哪些文件、函数和数据？
- 哪些业务规则决定最终结果？
- 现有测试证明了什么，又遗漏了什么？
- AI 修改代码后，为什么可以接受，或者为什么必须拒绝？
- 仍有哪些风险需要人工审批、监控或后续测试？

可以选择自己的垂直项目、课程提供的示例仓库或开源项目。公开提交时只保留代码片段、图表和去敏化记录，不提交密钥、本地绝对路径、客户数据或私有源码。

## 项目边界

- 不考察从零手写 Python 的速度
- 不考察复杂算法和语法记忆
- 不要求重构整个项目
- 只选择一条清晰的用户流程，通常涉及 3-8 个关键函数
- AI 可以生成分析草稿、测试和补丁，但最终判断必须由学员负责

推荐流程示例：上传训练视频 → 后端接收文件 → 姿态分析 → 生成报告 → 返回训练建议。

## 逐步关卡

| 关卡 | 学员任务 | AI 可以参与 | 通过检查 |
| --- | --- | --- | --- |
| 1. 定义行为合同 | 选择一条用户流程，写清用户目标、输入、正常结果、失败结果和不可接受风险 | 帮助把自然语言整理成 Given/When/Then | 至少 5 个具体样例，含正常、边界、失败和风控情形 |
| 2. 建立代码地图 | 找到入口文件、接口、核心函数、数据模型、外部服务和测试 | 搜索符号、解释文件、生成调用图草稿 | 每个箭头都能指向真实文件或函数；能口头走完主路径 |
| 3. 追踪执行与状态 | 对 5 组输入，在运行前预测分支、状态变化、输出和副作用，再与实际结果比较 | 生成追踪表模板、解释陌生语法 | 预测差异都有原因，不用“AI 说的”代替证据 |
| 4. 审阅测试合同 | 阅读现有测试，把断言翻译成业务语言，找出未覆盖的规则 | 总结测试、建议边界案例、生成补充测试 | 能说明每个测试证明什么、不能证明什么；补充至少 3 个高价值案例 |
| 5. 注入逻辑故障 | 让 AI 分别制造分支错误、状态错误和副作用错误；先预测影响，再运行测试和修复 | 生成受控故障与修复补丁 | 三种故障都有预测、现象、测试结果、根因和修复证据 |
| 6. 评审真实补丁 | 提出一个小型业务修改，让 Codex 实现；逐文件检查修改、影响范围和验证结果 | 完成实现、列出自检结果 | 给出接受/拒绝结论、证据、残余风险和回滚方式 |

## 三类必做逻辑故障

### 1. 分支或边界错误

示例：把“大于等于”改成“大于”、遗漏空列表、把错误状态当作成功返回。

检查重点：决策表是否完整，边界案例是否能失败。

### 2. 状态或数据错误

示例：覆盖已有字段、使用错误默认值、schema 接收了不该接收的数据、一次更新丢失历史状态。

检查重点：修改前后状态、数据来源、默认值和验证规则。

### 3. 副作用或重试错误

示例：重试时重复写入报告、重复发送通知、重复调用付费模型，或者失败后留下半成品。

检查重点：幂等性、重试次数、日志、回滚和人工审批点。

## 补丁评审清单

1. **需求：**这次修改真正解决哪个用户问题？
2. **范围：**改了哪些文件、接口、数据模型和共享函数？
3. **逻辑：**新增或改变了哪些条件、循环、默认值和状态？
4. **副作用：**会写文件、数据库，调用模型或发送消息吗？
5. **兼容：**旧数据、旧接口和旧页面是否仍然可用？
6. **测试：**哪些证据覆盖正常、边界、失败和风控路径？
7. **风险：**还有什么无法仅靠自动测试证明？
8. **结论：**接受、修改后接受，还是拒绝？为什么？

## 通用验收

- 能从用户界面或 API 入口追踪到核心 Python 逻辑和输出
- 代码地图中的节点和箭头都有可核对的依据
- 每次运行前先留下预测，不在看到结果后倒推答案
- 至少发现一个现有测试没有覆盖的业务逻辑风险
- 三类逻辑故障都能被复现、解释和修复
- 对 Codex 补丁给出明确的接受或拒绝决定
- 结论由测试、日志、样例或代码位置支持
- 能区分“测试通过”和“产品可以安全交付”
- 公开材料通过隐私检查，不包含本地路径、密钥和私人数据

## AI 使用规则

AI 在本项目中是开发执行者和分析助手，不是最终责任人。

可以让 AI：

- 解释 Python 语法、文件职责和调用关系
- 搜索相关代码并生成代码地图草稿
- 生成受控的故障补丁和补充测试
- 实现小型需求并列出自检结果
- 扮演评审者提出反例

学员必须：

- 在调用 AI 前写出业务目标和验收标准
- 在运行前预测代码行为和影响范围
- 逐项核对 AI 对文件、函数和数据的描述
- 决定接受、修改或拒绝补丁
- 亲自运行验证并保存证据
- 在关闭 AI 后完成口试

禁止把 AI 的自然语言总结直接当作证据。

## 完成证据

1. 一页业务行为合同
2. 一张代码与数据流地图
3. 五条执行轨迹记录
4. 一份测试合同与覆盖缺口表
5. 三份逻辑故障实验记录
6. 一份 Codex 补丁评审报告
7. 一段 8-10 分钟项目演示
8. 一次 10-15 分钟反 bypass 口试记录

## 展示建议

演示时不要逐行讲代码。用以下顺序：

1. 用户遇到什么问题
2. 一次操作如何穿过系统
3. 最危险的逻辑错误在哪里
4. 现有测试为什么没有发现，或如何发现
5. Codex 做了什么修改
6. 你用什么证据决定是否交付
7. 下一步如何降低残余风险
