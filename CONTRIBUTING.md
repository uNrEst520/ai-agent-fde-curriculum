# 参与课程建设

课程欢迎学习者、工程师、产品经理、研究者和行业专家共同维护。

## 小型修正

错别字、失效链接、版本兼容说明等，可以直接提交 Pull Request。请说明检查日期和替代来源。

## 实质性课程变更

增加、删除或替换正式课程内容前，请先提交 Course Change RFC。提案至少回答：

1. 当前课程存在什么明确问题。
2. 影响哪个能力目标和哪些学习者。
3. 新方案覆盖什么，遗漏什么。
4. 是否有更简单的替代方案。
5. 如何通过作业或项目验证学习效果。
6. 是否完成过最小教学实验。

“这个工具很热门”不足以成为进入共同核心的理由。

## 推荐论文

请使用 Paper Proposal 模板。论文推荐需要包括：

- 原始论文链接和版本日期。
- 论文解决的问题与核心证据。
- 与课程能力目标的关系。
- 适合进入核心、专修、工具实验还是只进入观察列表。
- 可执行的复现或课堂实验。
- 已知限制、反例和安全风险。

## 学习者提问

提问时请说明：你正在完成哪个单元、已经尝试了什么、看到什么结果、具体卡在哪里。问题本身也可能成为下一版课程的环境说明或常见错误案例。
## 公开发布安全

提交前必须确认：

- 本机路径使用 `<PROJECT_ROOT>` 或 `<CURRICULUM_ROOT>` 占位符。
- 不提交真实姓名、个人邮箱、电话、地址和客户身份信息。
- 不提交 API 密钥、访问令牌、Cookie、私有仓库地址或 `.env` 文件。
- 截图和日志已移除用户名、目录、账号、设备名和业务数据。
- 示例数据已经虚构或脱敏，并且不能反推出真实主体。

详细规则见 [公开仓库隐私与发布安全](docs/governance/publication-safety.md)。


## 课程设计原则

- 能力目标优先于工具名称。
- 原始来源优先于二手总结。
- 作业和评测优先于被动观看。
- 真实业务结果优先于只可演示的 Demo。
- 先修关系和认知负担必须明确。
- 自动发现可以广，正式收录必须谨慎。
