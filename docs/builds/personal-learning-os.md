# B00 · Build Your Own 个人学习操作系统

- **对应课程：**[O00](../curriculum/modules/00-learning-operating-system.md)
- **建议投入：**12-20 小时
- **难度：**入门
- **学习方法：**每一关都完成“构建 → 验证 → 解释”，再进入下一关

## 最终要造什么

一个版本化证据库，能记录概念、系统图、AI 决策、失败、口试和复盘。

## 第一遍约束

第一版只用 Markdown、目录和 Git，不引入笔记数据库或自动摘要。

这些约束只用于暴露原理。通过核心验收后，应与成熟库或框架对照，不建议把教学实现直接用于高风险生产。

## 逐步关卡

| 关卡 | 亲手构建 | 通过检查 |
| --- | --- | --- |
| 1. 定义证据 | 设计 concept、build、failure、review 四类记录及必填字段。 | 随机抽一条记录，能回答学了什么、证据在哪里。 |
| 2. 建目录 | 创建 inbox、concepts、builds、failures、reviews 和 templates。 | 新学习任务能在 2 分钟内找到归档位置。 |
| 3. 记录 AI 使用 | 为每次关键 AI 协作记录请求、建议、采纳/拒绝和验证。 | 至少有一条拒绝 AI 建议的可解释记录。 |
| 4. 建口试卡 | 每个概念生成机制、反例、取舍三类问题，但答案亲自写。 | 闭卷回答后能用原始证据纠错。 |
| 5. 建周复盘 | 统计投入、构建、失败、遗留问题和下周风险。 | 复盘能改变下一周任务，而非只总结。 |
| 6. 做理解审计 | 随机抽取旧项目，重画数据流并修复一个小故障。 | 导师能从证据链判断“运行”与“理解”的差距。 |

## 通用验收

- 从空目录或最小脚手架开始，提交每一关的独立 commit。
- 正常、边界、错误和故障案例都有自动测试或可重复实验。
- 能画出输入、状态、处理、输出和失败路径。
- 能闭卷解释一个关键算法或协议，并现场修改一个小行为。
- 记录 AI 提出的关键建议、人工采纳/拒绝理由和最终验证。
- 与一个成熟实现比较正确性、性能、功能边界和维护成本。

## AI 使用规则

可以让 AI 解释概念、生成测试想法、审查代码和定位错误；不得让 AI 一次性生成最终仓库后直接提交。每关先写自己的接口、假设和验收，再让 AI 参与实现。

## 完成证据

1. 代码仓库与阶段提交。
2. 架构/算法说明。
3. 测试或实验报告。
4. 至少三个失败样本。
5. 成熟实现对照。
6. 10-20 分钟口试记录。
