# B04 · Build Your Own 进程、网络与缓存实验室

- **对应课程：**[SE04](../curriculum/modules/04-computer-systems-networks.md)
- **建议投入：**35-50 小时
- **难度：**基础
- **学习方法：**每一关都完成“构建 → 验证 → 解释”，再进入下一关

## 最终要造什么

一个能观察进程资源、传输请求、处理并发、缓存结果和注入故障的系统实验室。

## 第一遍约束

关键协议和缓存先用标准库实现，不把全部行为交给容器或云平台隐藏。

这些约束只用于暴露原理。通过核心验收后，应与成熟库或框架对照，不建议把教学实现直接用于高风险生产。

## 逐步关卡

| 关卡 | 亲手构建 | 通过检查 |
| --- | --- | --- |
| 1. 观察进程 | 记录 PID、父子关系、CPU、内存、环境和打开文件。 | 能解释程序从命令到进程的变化。 |
| 2. 实现 TCP 协议 | 定义消息边界、编码、请求 id 和错误。 | 分包、粘包和断连测试可复现。 |
| 3. 加超时重试 | 实现 deadline、有限重试和退避。 | 证明重试不会造成无限重复操作。 |
| 4. 建缓存 | 实现 key、TTL、淘汰、命中和失效。 | 测试过期、脏数据和并发读写。 |
| 5. 注入并发故障 | 制造竞态、死锁或资源耗尽并加观察指标。 | 能定位而非只重启。 |
| 6. 容器化对照 | 加入健康检查、资源限制和网络配置。 | 说明容器解决与未解决的问题。 |

## 通用验收

- 从空目录或最小脚手架开始，提交每一关的独立 commit。
- 正常、边界、错误和故障案例都有自动测试或可重复实验。
- 能画出输入、状态、处理、输出和失败路径。
- 能闭卷解释一个关键算法或协议，并现场修改一个小行为。
- 记录 AI 提出的关键建议、人工采纳/拒绝理由和最终验证。
- 与一个成熟实现比较正确性、性能、功能边界和维护成本。

## AI 使用规则

可以让 AI 解释概念、生成测试想法、审查代码和定位错误；不得让 AI 一次性生成最终仓库后直接提交。每关先写自己的接口、假设和验收，再让 AI 参与实现。

## 完成证据

1. 代码仓库与阶段提交。
2. 架构/算法说明。
3. 测试或实验报告。
4. 至少三个失败样本。
5. 成熟实现对照。
6. 10-20 分钟口试记录。
