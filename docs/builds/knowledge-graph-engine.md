# B28 · Build Your Own 知识图谱与推理引擎

- **对应课程：**[XKG](../curriculum/electives/knowledge-graphs-neurosymbolic.md)
- **建议投入：**50-80 小时
- **难度：**高级选修
- **学习方法：**每一关都完成“构建 → 验证 → 解释”，再进入下一关

## 最终要造什么

一个具备三元组、schema、查询、规则推理、来源和图 RAG 的小型知识系统。

## 第一遍约束

先用基础图结构实现核心查询和推理，再接图数据库。

这些约束只用于暴露原理。通过核心验收后，应与成熟库或框架对照，不建议把教学实现直接用于高风险生产。

## 逐步关卡

| 关卡 | 亲手构建 | 通过检查 |
| --- | --- | --- |
| 1. 三元组存储 | 设计 entity、relation、value/id、source 和时间。 | 重复、冲突和身份合并可处理。 |
| 2. Schema 与约束 | 定义类型、domain/range、唯一和必需关系。 | 错误事实在写入时报告。 |
| 3. 图查询 | 实现邻接、路径、模式匹配和过滤。 | 查询结果可追踪到事实。 |
| 4. 规则推理 | 实现若干前向规则和派生事实来源。 | 循环规则不会无限推导。 |
| 5. 图与文本结合 | 做实体链接、证据子图和图 RAG 上下文。 | 答案引用到节点、边和原文。 |
| 6. 对照评测 | 比较文本 RAG、图查询和混合方案。 | 正确性、解释性、更新和成本分开评。 |

## 通用验收

- 从空目录或最小脚手架开始，提交每一关的独立 commit。
- 正常、边界、错误和故障案例都有自动测试或可重复实验。
- 能画出输入、状态、处理、输出和失败路径。
- 能闭卷解释一个关键算法或协议，并现场修改一个小行为。
- 记录 AI 提出的关键建议、人工采纳/拒绝理由和最终验证。
- 与一个成熟实现比较正确性、性能、功能边界和维护成本。

## AI 使用规则

可以让 AI 解释概念、生成测试想法、审查代码和定位错误；不得让 AI 一次性生成最终仓库后直接提交。每关先写自己的接口、假设和验收，再让 AI 参与实现。

## 完成证据

1. 代码仓库与阶段提交。
2. 架构/算法说明。
3. 测试或实验报告。
4. 至少三个失败样本。
5. 成熟实现对照。
6. 10-20 分钟口试记录。
