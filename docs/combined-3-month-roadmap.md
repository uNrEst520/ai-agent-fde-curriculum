# 3 个月双轨转型计划：AI Builder + 可解释工程能力

## 总目标

3 个月内形成两种同时成立的能力：

1. 能向面试官或创始人讲清楚：一个 AI/Agent 产品如何从业务问题走到可试用 Demo、数据闭环和商业化路径。
2. 能在上岗后用 Codex 做全栈开发，并且不是 bypass 状态：能理解系统结构、读懂关键代码、定位问题、提出验收标准和控制风险。

这条路线不把你训练成传统程序员，而是训练成 **会用 AI 工具链交付产品的 AI Builder**。

## 核心原则

- 作品集是主战场：继续围绕网球 AI 产品矩阵，而不是从零另开一堆练习项目。
- 基础学习服务于项目理解：学到 API、状态、数据结构、测试、部署时，马上回到现有项目里找对应代码。
- 每周必须做“反 bypass”练习：不要求手写全部代码，但要能解释本周新增功能的数据流、关键函数、失败模式和测试结果。
- 面试表达和工程能力同步训练：既能讲商业价值，也能讲实现逻辑。

## 第 1-2 周：建立工程地图

学习目标：

- 理解 Web 产品基本结构：前端、后端、API、数据库/文件存储、部署。
- 理解 JavaScript/TypeScript 和 Python 的最小可用语法。
- 理解 Git、依赖、环境变量、命令行、构建和测试。

项目任务：

- 给 `网球视频识别` 画出系统地图：React 页面、FastAPI 接口、MediaPipe 分析、报告生成、导出文件。
- 跑通主项目验证命令，并记录每个命令验证了什么。
- 选 5 个核心文件做读码笔记：
  - `src/App.jsx`
  - `backend/app/main.py`
  - `backend/app/analyzer.py`
  - `backend/app/report.py`
  - `backend/tests/test_report.py`

反 bypass 验收：

- 能解释一次视频上传后，数据如何从前端进入后端，再变成报告。
- 能说清 `training_prescription` 为什么属于 Agent 决策层，而不是普通文案。
- 能独立修改一个文案字段或报告字段，并跑测试确认没坏。

## 第 3-6 周：掌握 AI 应用工程主干

学习目标：

- HTTP API、JSON 数据结构、前后端状态流。
- Prompt / 规则 Agent / RAG / 工具调用的区别。
- AI 输出如何做约束、评估和回归测试。
- 最小数据库概念：用户、记录、任务、报告、知识源。

项目任务：

- 把主项目报告页作为第一作品，完善演示流程：上传样例视频 -> 生成报告 -> 查看训练处方 -> 导出 Markdown。
- 把 `网球训练计划` 的训练库逻辑梳理成“Agent 可调用知识/规则模块”说明。
- 写一份“从比赛问题到训练处方”的数据流说明。
- 准备 20 条典型输入，用来人工评估训练建议是否具体、可执行、不过度承诺。

反 bypass 验收：

- 能解释一个 API 的输入、输出、错误情况和前端如何消费。
- 能解释规则 Agent 与 LLM Agent 的取舍。
- 能指出当前项目哪些地方不应该吹成 CV 算法能力。

## 第 7-9 周：从 Demo 走向商业化产品

学习目标：

- 产品需求拆解、MVP 范围控制、数据闭环、成本控制。
- 登录、存储、权限、支付、部署、监控这些商业化产品的基本模块。
- 如何用 Codex 拆任务、审代码、补测试、做回归。

项目任务：

- 设计“网球 AI 教练 SaaS / 小程序商业化版本”的 v1 范围。
- 说明哪些能力来自现有项目，哪些需要新开发。
- 给主项目补一份部署与演示清单。
- 做一页创始人视角材料：目标用户、痛点、付费场景、竞品差异、下一步验证。

反 bypass 验收：

- 能把一个模糊需求拆成页面、接口、数据、测试和验收标准。
- 能指出 Codex 生成代码后的 5 个检查点：数据流、异常、测试、体验、安全。
- 能说明如果上岗第一周接手一个 AI 产品，你会如何摸清系统。

## 第 10-12 周：求职与上岗准备

学习目标：

- 面试叙事、项目复盘、技术问答、现场拆需求。
- Agent 工程岗位常见能力：API 集成、工具调用、记忆/知识库、评估、部署。
- 创业公司协作方式：快速验证、可演示、可迭代、成本敏感。

项目任务：

- 完成作品集首页、项目一页纸、Agent 架构图、面试故事。
- 准备 6 个项目故事和 20 个技术问答。
- 用 Codex 模拟一次“创始人给需求 -> 你拆方案 -> 你实现 MVP -> 你验收”的流程。
- 开始投递：Agent 工程师、AI 应用工程师、AI Product Engineer、AI Workflow Engineer、AI Builder。

反 bypass 验收：

- 3 分钟讲清产品价值。
- 5 分钟跑完 Demo。
- 10 分钟讲清架构、数据流、Codex 工作方式和下一步商业化。
- 面试官追问关键代码时，能定位到文件和函数，讲清大概逻辑。

## 每周固定节奏

每周 10-15 小时建议这样分配：

- 3 小时：基础概念学习，只学和当前项目有关的内容。
- 4 小时：读现有项目代码，画数据流，做读码笔记。
- 4 小时：用 Codex 实现一个小功能或修一个小问题。
- 2 小时：跑测试、写复盘、准备面试表达。

## 反 bypass 工作法

每次让 Codex 开发功能后，都要补这 6 件事：

1. 用自己的话写一句：这个功能解决什么用户问题。
2. 画出数据流：输入、处理、输出、存储。
3. 找到 3 个关键函数或组件。
4. 记录 2 个可能失败的情况。
5. 跑至少 1 个验证命令。
6. 写 5 行复盘：我理解了什么、还不理解什么、下次要问什么。

## 上岗能力清单

3 个月后你不需要变成传统全栈程序员，但应该具备：

- 能读懂一个中小型 AI Web 项目的目录结构。
- 能理解前端状态、后端 API、JSON 数据和测试的关系。
- 能用 Codex 拆任务、改代码、跑测试、定位报错。
- 能判断 AI 输出是否可靠，知道如何加约束和验收。
- 能把业务需求翻译成 Agent 流程、页面、接口和数据结构。
- 能向创始人解释技术取舍、成本、风险和下一步验证。

## 不做什么

- 不按传统算法工程师路线深挖数学和模型训练。
- 不追求从零手写复杂框架。
- 不把四个项目强行合并成大平台。
- 不把所有 AI 建议都包装成大模型能力；能规则解决的先规则解决。
- 不继续完全 bypass：每周必须留下读码笔记、数据流图和验证记录。
