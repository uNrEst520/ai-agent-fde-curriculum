# B14 · Build Your Own 可恢复 Agent Runtime

- **对应课程：**[AG01](../curriculum/modules/14-agent-runtime.md)
- **建议投入：**55-85 小时
- **难度：**高级
- **学习方法：**每一关都完成“构建 → 验证 → 解释”，再进入下一关

## 最终要造什么

一个有显式状态、工具、预算、检查点、人工接管和轨迹评测的 Agent 运行时。

## 第一遍约束

核心循环不使用 Agent 框架；所有状态转换必须可观察和可重放。

这些约束只用于暴露原理。通过核心验收后，应与成熟库或框架对照，不建议把教学实现直接用于高风险生产。

## 逐步关卡

| 关卡 | 亲手构建 | 通过检查 |
| --- | --- | --- |
| 1. 状态机 | 定义任务、消息、计划、工具结果、状态和终止原因。 | 非法状态转换被拒绝。 |
| 2. 决策循环 | 实现 observe、decide、act、record 与最大步数。 | 任务不会无限运行。 |
| 3. 工具边界 | 接入工具注册、schema、权限、超时和副作用确认。 | 工具失败有明确状态。 |
| 4. 记忆策略 | 区分工作状态、长期记忆、检索和写入审核。 | 错误记忆可定位和删除。 |
| 5. 检查点恢复 | 持久化事件，支持暂停、恢复、重试和人工修改。 | 进程中断后不重复已确认动作。 |
| 6. 轨迹评测 | 记录每步输入、选择、成本和结果，比较固定工作流。 | 证明何时 Agent 值得使用。 |

## 通用验收

- 从空目录或最小脚手架开始，提交每一关的独立 commit。
- 正常、边界、错误和故障案例都有自动测试或可重复实验。
- 能画出输入、状态、处理、输出和失败路径。
- 能闭卷解释一个关键算法或协议，并现场修改一个小行为。
- 记录 AI 提出的关键建议、人工采纳/拒绝理由和最终验证。
- 与一个成熟实现比较正确性、性能、功能边界和维护成本。

## AI 使用规则

可以让 AI 解释概念、生成测试想法、审查代码和定位错误；不得让 AI 一次性生成最终仓库后直接提交。每关先写自己的接口、假设和验收，再让 AI 参与实现。

## 完成证据

1. 代码仓库与阶段提交。
2. 架构/算法说明。
3. 测试或实验报告。
4. 至少三个失败样本。
5. 成熟实现对照。
6. 10-20 分钟口试记录。
